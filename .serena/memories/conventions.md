# Code Conventions

## Testing / Stubs
- `OllamaClient.call()` supports stub mode: set `System.setProperty("ollama.stub", "response text")` to bypass the real LLM call
- Template in `tools/langgraph_to_proto.py` at `generate_ollama_client_java()` — stub check is always generated
- Tests that don't need an LLM can set this in `setUp()` and clear in `tearDown()` with `System.clearProperty("ollama.stub")`

## Java
- `max_tokens` in Python `_sample(system, user, max_tokens=N, agent_name=...)` is now propagated to generated Java as `sample(system, user, N)`. The codegen translates `max_tokens` keyword into a `int maxTokens` param on the generated `sample()` method, which calls `.setMaxTokens(maxTokens)` on `CreateMessageRequest.Builder`.
- `BaseAgentNode.createMessage()` fallback: when `max_tokens` is not set (0), defaults to 2048 instead of 4096.
- Prompt engineering: constrained prompts drastically reduce LLM latency. Use format/length constraints in system prompts (e.g. "Reply with ONE word", "Write 3-5 sentences"). Avoid open-ended instructions like "Explain" or "Write a summary" — they produce verbose completions. Set `max_tokens` per call: routing/query calls need 16-256, generation calls need 1024-2048.
- `OllamaClient`: HTTP client has 10s connect timeout, 2 retry attempts (down from 3), 500ms base delay (down from 1000ms).

## Java
- Package: `research.v1` (all sources, tests, generated code)
- Build output directory: `gen/` (Java stubs in `gen/java/`, compiled classes in `gen/classes/`, `gen/main-classes/`, `gen/test-classes/`)
- No comments in generated code except the one-line `// Auto-generated by langgraph_to_proto.py` header — the generator strips comments
- JavaDoc only on manually written tool classes (e.g. `WebSearchTool.java`), not on generated wrappers
- No Lombok, no annotation processors — plain Java with manual constructors/getters
- gRPC interceptor pattern: `AgentIdInterceptor` (implements `ServerInterceptor`) with `AGENT_ID` and `SERVER_NAME` context keys
- MCP server implementations extend generated gRPC service base classes (`MCPServerServiceImplBase`)
- Agent nodes extend `BaseAgentNode` (which extends `MCPClientServiceImplBase`)
- Tool classes (`WebSearchTool`, `FilesystemTool`) are standalone — no gRPC dependencies, usable from any transport adapter
- `impl/*_impl.java` files contain only the body of `CallTool` handler methods (methods + constructor), not full class — they get merged into generated `*MCPServerImpl.java` wrappers
- Static test helpers: `section(title)`, `print(label, value)`, `divider()`, `agentHeaders(agentId)`

## Python (code generation)
- LangGraph: `StateGraph` with `add_node`/`add_edge`/`add_conditional_edges`
- Agent functions use `@tool` decorator for tool definitions
- State class: `TypedDict` with `Annotated` fields (list messages use `add_messages` reducer)
- Type annotations required — AST-based translator uses them

## JSON-RPC
- Hand-implemented protocol (Gson + `com.sun.net.httpserver`), no third-party library
- Version field strictly validated: `"jsonrpc": "2.0"` required; wrong version → 400, invalid JSON → -32700
- Notifications: requests with no `id` field → HTTP 202, empty body
- HTTP path: `/mcp` (e.g. `http://localhost:50061/mcp`)
- Agent node fallback: try `initHttpServer()` first (JSON-RPC HTTP), if unavailable fall back to gRPC stub
- `MCP-Protocol-Version: 2025-11-25` header, `Accept: application/json` or `text/event-stream`

## Proto
- Generated output: `proto/research/v1/research_agent.proto`
- Buf config: `buf.yaml` (module), `buf.gen.yaml` (codegen plugins)

## File Patterns
- Generated files: marked `// Auto-generated by langgraph_to_proto.py`, stripped of comments
- Manual Java: `src/main/java/research/v1/*.java` and `src/main/java/research/v1/impl/*.java`
- Tests: `src/test/java/research/v1/*.java`
